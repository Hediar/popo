- **formatContent 개선**: `[project] POPO AI` 형태로 타입 표시 + metadata 포함

## 발견한 문제 / 고민

### 고민 1: 입사일 Hallucination
- "입사일이 언제예요?" → career 벡터 검색 유사도 0.6 미만으로 결과 0건
- AI가 프로필의 "2년 4개월"만 보고 날짜를 추측 → **2021년 8월이라고 거짓 답변**
- 타입 보충 로직으로 해결했지만, 벡터 유사도 임계값(0.6)이 너무 높은 것 아닌가?
- career/experience 데이터가 각 1건뿐이라 벡터 검색보다 직접 조회가 나을 수도

### 고민 2: metadata의 duration → startDate/endDate 분리
- 원래 duration을 startDate/endDate로 분리하려 했으나 아직 미적용
- 엔티티에서 필드를 뺐으니, metadata JSON 내부에서만 관리할지 결정 필요
- SQL 업데이트 쿼리는 준비되어 있음 (대화 초반에 작성)

### 고민 3: CORS 문제 답변 오류
- "CORS 문제를 어떻게 해결했어요?" → FSMS 프로젝트인데 불도저 프로젝트라고 답변
- 벡터 검색에서 불도저가 먼저 잡히거나, 컨텍스트에 CORS 관련 내용이 여러 프로젝트에 있어서 혼동
- 검색 결과의 우선순위/정확도 개선이 필요할 수 있음

### 고민 4: 추상적 질문 답변 품질
- "강점이 뭐예요?", "가장 자랑하고 싶은 성과" 같은 질문
- 검색 결과에 의존하다보니, 벡터 유사도 높은 데이터 기준으로만 답변
- 실제로는 운영 업무 50% 절감, UI/UX 개선으로 오류 80% 감소 등이 더 임팩트 있는 성과
- experience 데이터의 achievements가 더 잘 활용되어야 함

### 고민 5: 환경변수 관리
- 로컬 개발 시 `.env` → Spring Boot가 자동 로드 안 함
- IDE Run Configuration에 매번 설정하거나, spring-dotenv 같은 라이브러리 도입 고려
- 배포(Render)에서는 환경변수로 주입되므로 문제없지만, 로컬 개발 편의성 개선 필요

### 고민 6: 토큰 비용 증가
- 검색 결과 5개 → 10~15개로 늘리면서 프롬프트 토큰 사용량 증가
- GPT-4o-mini라서 비용은 크지 않지만, 응답 시간도 늘어남
- 필요한 정보만 정확히 검색하는 것이 이상적 (검색 정확도 > 검색 수)
